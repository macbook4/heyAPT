# heyAPT Plan Spec (MVP)

## 1. 목표와 성공 기준
- 제품 목표: 감도 있는 인테리어 매물 POI를 지도 기반으로 빠르게 축적한다.
- 1차 KPI: `등록 POI 수` (출시 후 4주).
- 품질 핵심 지표: `주소 오표기율 5% 이하`.
- 타깃 사용자: 매매/전월세 수요자.

## 2. 출시 범위와 운영 정책
- 지역 범위: 수도권(서울+경기+인천).
- POI 등록 주체: 일반 회원 포함.
- 게시 정책: `승인 후 게시`.
- 검수 SLA: `영업일 48시간 이내`.
- 중복 판정: 초기에는 운영 수작업 기준(자동 판정은 후속 단계).

## 3. 제품/UX 결정 사항
- 플랫폼: 웹+앱(WebView 하이브리드) 동시 출시.
- 앱 구조: Next.js 웹 1개 + Capacitor 래퍼(iOS/Android).
- 지도: Naver Maps API(Web SDK) 사용.
- POI 등록 시작 UX: `주소 검색 후 등록`(아파트 단위 주소 입력 기준).
- 등록 필수 입력(초기): 주소, 가격, 거래유형, 사진 1장, 상세설명(최소 글자수 적용).
- 등록 고도화: 면적/층/관리비 등은 내부 데이터 매핑으로 자동 보강.

## 4. 기술 구현 스펙
- 백엔드/API: Next.js Route Handler 기반.
- 인증: Supabase JWT (소셜+이메일 병행).
- 권한 모델:
  - `guest`: 조회만 가능
  - `owner`: 본인 POI 생성/수정/삭제
  - `admin`: 전체 POI 운영 권한
- 기본 API:
  - `GET /api/pois` (bounds 기반 목록)
  - `GET /api/pois/{id}` (상세)
  - `POST /api/pois` (등록)
  - `PATCH /api/pois/{id}` (수정)
  - `DELETE /api/pois/{id}` (삭제)
- 에러 포맷: `{ code, message, requestId? }`.

## 5. 데이터/검수/품질 관리
- 등록 주소는 검수 단계에서 표준화(도로명/지번/좌표 정합 확인).
- 주소 오표기 감축 전략:
  - 등록 시 주소 검색 강제
  - 검수자가 지도 좌표 재확인 후 게시
  - 오표기 신고 접수 시 우선 재검수
- 중복 처리(초기): 운영자가 주소/사진/설명 대조 후 병합 또는 반려.

## 6. 보안/권리/컴플라이언스
- 사진 권리 정책: 업로드 시 `권리 보유/사용 허가` 동의 필수.
- 위반 대응: 신고 접수 → 블라인드 처리 → 검수 결론 반영.
- 시크릿 관리: Naver API 키/Supabase 키는 `.env`와 배포 시크릿만 사용.
- 감사 로그: POI 생성/수정/삭제 시 서버에서 `request_id`, `actor_user_id` 기록.

## 7. 트레이드오프 (의사결정 근거)
- `승인 후 게시` 선택:
  - 장점: 주소 품질과 신뢰성 확보.
  - 단점: POI 증가 속도 저하.
- `수도권 전체` 선택:
  - 장점: 공급 풀 확대.
  - 단점: 검수 운영비 증가.
- `주소 검색 후 등록` 선택:
  - 장점: 실제 아파트 단위 입력 UX 적합.
  - 단점: 좌표 오차 가능성(검수로 보정 필요).
- `중복 수작업` 선택:
  - 장점: 초기 오탐/누락 최소화.
  - 단점: 운영 확장성 낮음.

## 8. 테스트/수용 기준
- 기능:
  - 웹/앱에서 POI 등록→검수→게시 플로우 정상 동작.
  - bounds 변경 시 POI 조회/렌더링 정상.
- 권한:
  - guest 등록/수정/삭제 차단.
  - owner는 본인 리소스만 수정/삭제 가능.
  - admin은 전체 운영 가능.
- 품질:
  - 출시 후 4주 주소 오표기율 `<= 5%`.
  - 검수 SLA(48시간) 준수율 측정.

## 9. 후속 고도화 백로그
- 주소별 매핑 강제(검수 의존도 감소).
- 중복 자동 판정(좌표 반경+주소 유사도).
- 매물 속성 자동 보강(면적/층/관리비).
- 감도 점수 모델 도입(초기 데이터 축적 후).
